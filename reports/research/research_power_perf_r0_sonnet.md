## 0. 一句话结论

用户要的不是「把 `sustained_tflops` 的时序图换个横轴」，而是要**在同一功耗工况点上，看到跨样本/跨卡的性能分布**——这要求功耗是一个可重复独立采样的自变量，而现状（GEMM 满载 + 卡自身 DVFS 决定功耗）只给出**一条随时间单调演化的自相关轨迹**，直接分箱会把「还在爬坡的瞬态」和「已到热稳态的平台」混成一个假分布。**采集管道形状确实已基本齐备（同意 D1 核心判断），但必须先加一个零代码改动的「稳态门禁」，且被动漂移功耗天然会挤在功耗上限附近、动态范围很窄——這比 D1 估计的更容易在 16 卡冒烟就撞见，需要提前把「轻量功耗扫描」的能力检查（而不是决策）挪到 Phase 1。**

## 1. 第一性原理：power–perf 空间该怎么采样才有统计意义

### 1.1 先把用户诉求翻译成统计问题

用户说「不只时间维，要看每卡在不同功耗下的性能体现（分布曲线）」。翻译成统计语言：要估计条件分布 **p(perf | power, card)**，并且要能跨卡比较这个条件分布。这与「画 `tflops` vs `t_s` 的时间序列」是**两个不同的估计目标**：

| | 时间序列 | power–perf 分布 |
|---|---------|-----------------|
| 自变量 | 时间 `t_s`（代理变量，物理上不驱动性能，只是热爬升的时钟） | 功耗（更接近真实物理驱动量：功耗墙/DVFS 决定频率，频率决定算力） |
| 每个 x 值有几个 y 样本 | 恰好 1 个（一条轨迹） | 需要 ≥1 个独立样本，理想 ≥30 个才能谈分布/CV |
| 跨卡可比性 | 不可比——两张卡在同一 `t_s` 可能处在完全不同的热历史阶段 | 可比——只要功耗匹配，理论上是同一物理工况 |

**结论**：要从「一条轨迹」变成「一个分布」，缺的不是画图技巧，是**独立重复采样**。

### 1.2 采样设计的三个必要条件

1. **功耗轴要有真实动态范围，且最好是外生（exogenous）而非纯内生（endogenous）**
   现状 `sustained_tflops` 让 GEMM 全速跑、功耗完全由卡自己的功耗墙/温控决定——这是**内生**变量：功耗和算力同时被同一个隐变量（温度/时间）驱动，二者互为同源产物而非因果关系。拿内生 `power_w` 当 x 轴画「曲线」，本质上是在画「(功耗, 算力) 关于隐藏时钟的参数曲线」，不是「给定功耗、算力的响应函数」。真正有因果解释力的自变量是**外生刺激**：显式功耗上限（`power_limit_w` 主动设档，等价 `nvidia-smi -pl` / npu 侧的功率封顶接口）或负载占空比（GEMM 窗口间插入空闲，间接压低平均功耗，不改变频率/温度耦合关系）。这是本方案与 D1「合并 sustained、被动采集」的第一个分歧点（详见 §1.4、§7）。
2. **稳态门禁（steady-state gating）**
   `sustained_tflops` 的样本天然分两段：**爬坡段**（前几秒，频率/功耗随缓存/热容爬升，`d(power)/dt`、`d(temp)/dt` 明显非零）和**平台段**（进入热稳态后 `power_w` 趋于常数，多数贴着 `power_limit_w`）。这两段的物理意义完全不同——爬坡段的低功耗点是「还没到目标工况」，不是「该卡在低功耗下的真实表现」。若不区分，分箱统计会把爬坡段的低功耗样本错当成「低功耗稳态表现」，人为拉出一条不存在的「低功耗到高功耗性能上升」的假曲线（这条曲线的真实成因是热爬升，不是功耗响应）。**门禁规则建议**：以窗口级 `power_w` 环比变化率 + 温度环比变化率双双低于阈值（例如连续 3 个窗口 `|Δpower_w| < 2%` 且 `|Δtemp_c| < 1℃`）作为进入稳态的判据，之后的样本才计入 power–perf 分布；爬坡段单独打 `phase="ramp"` 标签，只进时间序列图，不进分布图。
3. **独立重复，而不是同一轨迹内的相邻样本**
   同一次烧机里相邻两个 window 的 `power_w`/`tflops` 高度自相关（共享同一热状态），把它们当独立样本会**系统性低估**方差、虚增「分布」的置信度。真正独立的重复应来自：跨卡（cluster cloud，卡与卡之间热历史、硅片差异是独立的）、跨 run（同一张卡不同时间点起跑，冷启动状态不同）、以及（如果做了 §1.4 的主动扫描）跨功耗档之间的重复测量。单次单卡烧机内部的「稳态段」样本可以合并成**该卡该次 run 在该功耗档的一个中位数点**，但不应把它的窗口数直接当作分布的自由度。

### 1.3 现有 `sustained_tflops` 几何形状对采样设计的直接含义

结合 `stage_a.py` 的实现：`sustained_tflops(seconds=30, window=50)` 在固定 `n=8192` GEMM 下连续压测，每 50 发 GEMM 记一个 `{t_s, iter_ms, tflops, ...TELEMETRY_KEYS}` 样本，其中 `_attach_telemetry` 在计时窗口**外**读取 `TelemetrySampler.latest()`（不扰动计时，设计上是对的）。这意味着：

- 一次 30s 烧机，`window=50`、`n=8192`、bf16 下单次 GEMM 耗时量级与遥测轮询间隔（NPU 默认 `interval_s=0.5`，无 `-lms`，只能轮询）大致同阶——**具体倍数需要机上实测**，但只要窗口耗时接近或短于轮询间隔，就会出现「连续几个 window 读到同一份缓存遥测值」的**量化/走样（aliasing）**现象：功耗轴不是连续的，而是阶梯状，且这一步的边界与真实功耗变化时刻并不对齐（有相位滞后）。这个问题 D1 §1.3 提到过量级相近，但没有把它当作需要主动缓解的假阳性机制来处理（见 §2.3）。
- 30s 在 GEMM 满载下，多数卡大概率在前几秒内完成频率爬坡并贴到 `power_limit_w`——也就是说**真正落在「稳态」的样本，功耗值会高度集中在功耗墙附近，动态范围很窄**；能提供「不同功耗下的表现」的低功耗点几乎全部来自爬坡段（此时还不稳态，见 §1.2.2）。这是本方案认为需要在 Phase 1（16 卡冒烟）就显式检查、而不是等到「发现退化再评估」的原因：**从纯物理直觉出发（DVFS 系统的常见行为），被动漂移几乎必然给出退化的功耗动态范围，这不需要等数据验证才能预判**，提前规划比事后补救成本更低。

### 1.4 `power_w`（内生）vs `power_limit_w`（外生上限）：该把哪个当自变量

现状 `power_limit_w` 对同型号卡应是一个近乎常数的出厂/驱动配置值（除非机器已经手工设过功率墙），**如果它在样本里恒定，就不能拿它当自变量**，只能退回 `power_w` 做条件分布（观测性、非因果）。若要画「真正的功耗–性能响应曲线」（用户「不同功耗下的性能体现」这句话字面意义上更接近这个诉求），**理想方案**需要主动改变 `power_limit_w`（2–4 档即可）并在每档等待到稳态后测 `tflops`——这才是外生刺激、曲线上每个点是真正可比的因果响应，而不是被动漂移曲线上互相混杂着热历史的散点。这是本方案与 D1 的第二个分歧点：D1 把「独立 `power_ramp`」整体归为 Phase 3+、按需触发；本方案认为至少**能力检查**（这批 NPU/驱动是否支持设置功率上限、需要什么权限）应该现在做，不要等 16 卡冒烟"发现退化"才第一次去查有没有这个接口——真正的决策（要不要跑）可以留到冒烟后，但「能不能跑」的调研不应该被延后。

## 2. 假阳性清单

| # | 机制 | 说明 | 缓解 |
|---|------|------|------|
| 1 | **节点散热差异** | 同型号卡若所在节点风道/液冷更差，会在更低功耗时就先触发热降频——曲线整体左移/下压，看起来像「体质差」，根因是 host 环境。 | 复用 R3 的 `within_host` 残差思路：先按 host 分组算「同功耗档下相对本机中位数」的残差，再跨 host 比较；host 应作为随机效应而非直接忽略。 |
| 2 | **争用** | 后台/邻居进程抢占 PCIe/HCCL 带宽或 host CPU（影响 launch），并不总是在 preflight 时就存在，可能在烧机中途才出现。`sustained_tflops` 目前已有 `timing_flatline`/`perf_contended` 判据，但只看 `iter_ms` 是否异常"死平"，不看功耗侵入。 | 每个稳态窗口除了现有 flatline 检测，额外检查该窗口 host 侵入信号（若可拿到 host 负载/其它探针占用标记）；被标记的窗口从 power–perf 分布中剔除或单独打标，不与"干净"窗口混池。 |
| 3 | **遥测毒数据（本 Track 特有的两种新机制）** | (a) `power_w` 与 R3 已定性的 `temp_c≡2` 走的是**同一条** `npu_smi_info(device, "power")` → `parse_npu_power` 解析路径，存在同构中毒风险（命令语法/寻址错位一旦复发，`power_w` 会和 `temp_c` 一样退化成某个固定值）；一旦如此，任何 `tflops/W` 效率计算都是在拿近似常数做分母，算出来的"效率分布"只是 `tflops` 分布的重新缩放，毫无功耗信息量，但外观上仍像一张正常的图，极具迷惑性。(b) §1.3 的遥测轮询-窗口时长走样：连续窗口读到同一缓存功耗值，人为制造"同一功耗下算力有方差"（真实是"同一功耗读数覆盖了多个不同算力窗口"，是采样分辨率问题不是硅片方差）。 | (a) 上任何 power–perf 图前，先跑一次集群级 `power_w` 方差检查（同 R3 对 `temp_c` 的零方差红旗），零方差或方差异常小于测量噪声地板时整批标 `telemetry_untrusted`，拒绝出图。(b) 报告里同时给出"该功耗样本对应的遥测时间戳与窗口时间戳差值"分布；差值过大（比如超过遥测间隔一半）的样本弱化权重或剔除;更根本的缓解见 §4 的 `window` 参数调整建议。 |
| 4 | **workload 混淆** | 若功耗轴的变化是通过改 GEMM 形状/精度间接造成的（而不是显式功耗上限或占空比），性能差异里会混入"计算强度变了"和"功耗变了"两个因子，无法归因。 | power–perf 采样必须固定 shape/dtype（沿用现有 `sustained.n`/`dtype`），只让功耗通过温控自然漂移或主动 cap 变化；shape 扫描是另一个正交维度（`shape_sweep`），不要在同一批数据里混着比较。 |
| 5 | **幸存者偏差** | `preflight.require_idle` 会在启动前直接 abort 被判"不干净"的卡，这些卡不会贡献任何 power–perf 样本；如果这批卡恰好是散热/功耗有问题的卡，最终分布会系统性偏"好"。 | 报告注明 abort 卡数量/host 分布，不要把"参与烧机的卡的分布"直接等同于"全fleet的分布"。 |
| 6 | **共享供电域** | 同一 host 多卡若共享 PSU/供电轨，多卡同时顶到功耗上限时可能出现轨级限流，导致同 host 内多张卡的功耗曲线互相耦合（不是各卡独立表现）。 | 记录同 host 内并发跑 sustained 的卡数量/时间重叠情况；若发现同 host 卡的功耗曲线明显同步抖动，需要在报告里单独说明，不要归因到单卡体质。 |

### 4.2 理想方案（新增探针，能力检查现在做、决策可以留到冒烟后）

| 改动 | 说明 |
|------|------|
| 能力检查（**现在做，不阻塞冒烟**） | 调研这批 NPU 驱动/`npu-smi` 是否暴露设置功率上限的接口（等价 `nvidia-smi -pl`），需要什么权限（root/特权容器）；CUDA 侧作为对照组理论上有 `nvidia-smi -pl` 可用，可先在 CUDA 卡上验证整条 pipeline，NPU 侧再谈是否可行。 |

## 5. 与 constitution 探针如何挂接

- **零改动挂接点**：`sustained_tflops`（Stage A，`order=30`）本身已经是"烤机"探针，`func_perf`/`hbm_bandwidth` 的逐 round 遥测也可以贡献峰值工况的功耗–性能点云（补充稳态曲线之外的"瞬时峰值"参照点，但要注意峰值点通常在功耗尚未爬满时测得，不能和稳态点混池比较）。
- **门禁复用**：功耗轴的可信度判定应该并入 R2.5/R3 已经定义的 `telemetry_trust` 门禁，而不是另起一套——`slow_cause.py::classify()` 已经在处理 `temp_c` 零方差/毒数据的拒绝逻辑，`power_w` 的零方差红旗应该用同一个机制表达（同一个 `telemetry_trust=false` 状态即可屏蔽热/功耗/效率三类结论，不需要新增一个平行的"power_trust"标志）。
- **与 throttle 标志的交叉验证**：`TELEMETRY_KEYS` 里已有 `throttle_power_cap`/`throttle_hw_thermal`/`throttle_sw_thermal`（目前 NPU 侧这三个字段在 `NpuSmiProvider` 里永远是 `None`，见 `telemetry.py` 注释"npu-smi 暴露不出 clocks_event_reasons 等价物"）——这是本 Track 的一个隐藏阻塞项：**没有这三个标志，A3 图和 E1 的"膝点是不是热墙"这类归因就只能靠功耗–温度散点的形状推测，无法像 CUDA 侧一样直接读驱动给的因果标签**。如果后续要对 NPU 侧的功耗曲线做"是热墙还是功率墙"的归因，需要评估 `npu-smi` 是否有等价字段（例如健康/board 快照里可能带有降频原因位），这个调研本身可以现在做，不依赖遥测命令修复。
- **落地位置**：新增的 `phase`（ramp/steady）打标、`tflops_per_watt` 派生、遥测延迟诊断都建议放在报告侧（`gen_card_constitution_report.py` 或拆出的 `plot_power_perf.py`），保持"探针只管采集、报告管解释"的现有架构分层（`sustained_tflops` 不应该反过来知道"稳态判据"这种报告层概念）。理想方案的主动功率扫描才需要新探针 + `builtin.py` 注册 + config 开关，遵循 `gemm_shape_sweep` 已经确立的"混合停止条件 + `verdict_neutral=True` + 默认关"模式，不需要发明新范式。

## 8. Top 5

1. **现有 `sustained_tflops` 已经逐窗口落库 `power_w`/`power_limit_w`/温度/频率/throttle 标志，理论上"零新探针"就能画出观测性的 power–perf 图**——这一点与 D1 一致，是本 Track 目前最确定的好消息：缺的是报告层的分析逻辑，不是采集层的代码。
2. **但被动漂移功耗（不主动设功率上限，只让 GEMM 满载自然升温）在物理上大概率给出"退化"的功耗动态范围**——多数稳态样本会挤在功耗墙附近，能覆盖"不同功耗档"的点几乎全来自还未到稳态的爬坡段。不区分 ramp/steady 直接分箱，会把"卡还在热爬升"误读成"该卡在低功耗下性能就是这样"，制造一条不存在的假曲线。这是本方案与 D1 最大的实质分歧：D1 把这当作"冒烟后再评估"的可能性，本方案认为这是可以现在从第一性原理预判、应该提前规划能力检查的必然性。
3. **`power_w` 与 R3 已定性的 `temp_c≡2` 走的是同一条 `npu-smi` 解析路径，存在完全同构的中毒风险**——上任何功耗图之前，必须先对 `power_w` 做集群级零方差检查（复用 R3 对 temp 的红旗机制），否则算出来的 `tflops/W` 效率是在拿近似常数分母做除法，产出看似精确但毫无意义的数字。
4. **效率 (tflops/W) 不能跨卡直接比大小，必须先在同一功耗点上匹配**——效率本身随功耗非单调（常见先升后降），不同卡若被比较时实际运行的功耗档不同，比较结论会把"功耗档差异"误判成"体质差异"。这需要新增一张"匹配功耗点的效率分布"图（E2），D1 的图清单里没有覆盖这一步。
5. **节点散热/争用会通过"更早触发降频"系统性压低某些卡的功耗–性能曲线，看起来完全像体质差**——必须复用 R3 已经在做的 `within_host` 残差思路，把 host 当随机效应剥离（新增 H1 热力图），否则功耗维度的"换卡"结论会重复 R3 里已经踩过的"节点问题误判成卡问题"的坑。

## 9. 与「只画 sustained 时序」的本质区别

时间序列（`tflops`/`power_w` vs `t_s`）回答的是**单卡单轨迹的退化检测**：这张卡随着烤机时间推移有没有变差、有没有明显的降频拐点。它的自变量（时间）只是一个代理时钟，物理上不驱动性能——两张卡在同一个 `t_s` 完全可能处在不同的热历史阶段，因此**时间序列在结构上无法跨卡比较，也无法回答"给定同一个功耗，谁的表现更好/更稳"**。它给出的是 N 个高度自相关的样本描述一条轨迹，不是"分布"，因为它在任何一个自变量取值上都只有恰好一个观测点，没有能力分离"该工况下的真实方差"和"沿着轨迹系统性漂移"这两种完全不同的东西。

power–perf 分布做的是**范式转换**：把不可跨卡比较的时间轴换成物理上真正驱动性能的功耗轴，并且要求在同一功耗点上有多个独立观测（跨卡、跨 run，理想情况下还要跨主动设定的功耗档），才能谈"分布"而不是"一条曲线"。这样才能把"在同一物理工况下，这批卡的算力表现有多分散"（真正的体质差异 / 效率差异）和"这张卡随时间怎么变化"（退化/降频检测，时序图仍然要保留，但只是辅证，不是主结论）明确分开——本质区别就是**自变量从不可比的时间换成可比的物理量，代价是必须多做一层稳态门禁和独立重复采样，否则"分布"只是时序图的伪装**。
