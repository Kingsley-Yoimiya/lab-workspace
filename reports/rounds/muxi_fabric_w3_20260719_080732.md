# Muxi Fabric Probe — W3.1/W3.2 端侧只读门禁

- 可见性run：`20260719_080732-w31-qos-visibility`
- counter差分run：`20260719_081238-w32-netdev-counter-delta`
- 集群 / 身份：`vc-c550-h3c-test` / `yinjinrun.p`
- 代表pod：master、worker0、worker45
- 原则：只读；不修改TC/DSCP/PFC/ECN/MTU/交换机；不安装工具

## 1. TC=128映射链：当前可证实到哪里

### 层1：环境变量被读取——已证实

三个代表pod的近期MCCL INFO日志均多次出现：

`MCCL_IB_TC set by environment to 128.`

`libmccl.so` strings也包含 `MCCL_IB_TC`。因此可以确认当前进程读取了该环境
变量，并把值解析为128。

### 层2：QP traffic class被设置——未证实

`libmccl.so`同时可见：

- `MCCL_IB_TC`
- `tclass`
- `ibv_modify_qp`
- `wrap_ibv_modify_qp`

这些是实现线索，但没有源码、反汇编调用链或运行日志证明：

`MCCL_IB_TC=128 → ibv_qp_attr.ah_attr.grh.traffic_class=128`

MCCL INFO也没有打印QP traffic class值。因此本轮不把“变量被读取”升级为
“QP已设置”。

### 层3：线上DSCP/ECN——未证实

没有可用的硬件抓包、SPAN/TAP、RoCE数据面tcpdump、RNIC等价遥测或交换机
队列数据。**禁止**仅按标准8-bit ToS把128换算成DSCP 32、ECN 00并声称是
线上实际值。

### 层4以后：交换机映射——未知

当前链条停在：

`MCCL_IB_TC=128被MCCL读取`

后续 `QP tclass → wire DSCP/ECN/PCP → trust → priority → PG/queue →
PFC/ECN/CNP → RNIC rate response` 均缺ground truth。

## 2. xscale端侧counter可见性

三个代表pod、四个xscale rail全部检查：

| 入口 | 结果 |
|---|---|
| `/sys/class/infiniband/xscale_*/ports/1/counters` | 0/12存在 |
| `.../hw_counters` | 0/12存在 |
| xscale专用counter文件 | 0 |
| perfquery | 不可用 |
| rdma | 不可用 |
| ethtool | 不可用 |
| ibstat / iblinkinfo / devlink | 不可用 |
| xscale / xscale_tool CLI | 不可用 |
| `/sys/kernel/debug` | 可读但顶层为空 |
| `/proc/driver` | 仅mlx5_core/rtc，无xscale计数 |
| `ibv_devinfo -v` | 只提供状态、MTU、GUID、link_layer，无流量/拥塞counter |
| `mx-smi --help` | GPU/HBM/PCIe/MetaXLink，无xscale RoCE counter |

`/opt/maca/etc/default-counters.csv`列出GPU、PCIe、MetaXLink、ECC/RAS等指标，
但没有xscale/RoCE bytes、packet、CE、CNP、PFC、retry、drop、FEC或symbol
指标。`mx-diagease`是GPU/PCIe/MetaXLink诊断工具，不提供本轮需要的RNIC
counter入口。

因此当前容器内无法差分：

- RoCE data bytes / packets
- retry / timeout
- CE / ECN-marked packets
- CNP sent/received
- PFC pause tx/rx/duration
- queue drop / buffer occupancy
- CRC / FEC / symbol error

## 3. 通用netdev counter刺激验证

唯一安全可读的候选是 `/sys/class/net/net1..4/statistics`。使用已验证的
worker62→master单向P2P：

- 16MiB
- warmup3 + iters10，共预期218,103,808 bytes payload
- single xscale_0 / GID5 / TC128 / VSWITCH1

before/after结果：

- 两端net1..4都只增加约168～316 bytes、4～6 packets；
- net1并未增加约218MB；
- net2..4出现相近微小增量；
- errors/dropped全部0增量。

结论：通用netdev统计可读，但在当前驱动/卸载路径下**不对RoCE数据payload
敏感**，不能作为fabric bytes/packets counter。没有可用数据面counter，
因此不再启动额外AllReduce刺激。

## 4. MCCL算法、协议、channel与QP参数证据

### 有明确“环境变量被设置”字符串

| 参数 | 证据 | 当前可安全做什么 |
|---|---|---|
| `MCCL_ALGO` | libmccl：`set by environment to %s` | 小规模能力门禁 |
| `MCCL_PROTO` | libmccl：`set by environment to %s` | 小规模能力门禁 |

厂商样例 `xccl.sh` 使用 `NCCL_ALGO=Ring`、`NCCL_PROTO=Simple`；libmccl还包含
`Tree`、`Simple`、`LL128`字符串，并明确提示CollNet可能不支持而被忽略。

### 库可识别、但值域/生效未验证

- `MCCL_MIN_NCHANNELS`
- `MCCL_MAX_NCHANNELS`
- `MCCL_NCHANNELS_PER_NET_PEER`
- `MCCL_P2P_NCHANNELS*`
- `MCCL_IB_QPS_PER_CONNECTION`

现有INFO日志能看到默认生成32个channel和Trees，但这不证明上述变量的值域、
运行时接受或QP数量实际改变。本轮没有运行这些未知参数。

## 5. 下一步可安全A/B

按风险排序：

1. **参数控制能力门禁**：2节点或world64短任务，对比默认与显式
   `MCCL_ALGO=Ring / MCCL_PROTO=Simple`；只有INFO确认变量被接受且结果有效，
   才进入性能比较。
2. **算法小规模A/B候选**：Ring/Simple对Tree/Simple。Tree仅有库字符串证据，
   应先2节点能力门禁；不支持即停止。
3. channel/QP A/B暂缓。先向厂商索取
   `MCCL_MIN/MAX_NCHANNELS`、`MCCL_IB_QPS_PER_CONNECTION`的值域、默认值和
   生效日志，再决定1/2/4等扫描。
4. 不做TC数值A/B，直到QP tclass和交换机映射链得到确认。

## 6. 网络侧最小ground truth

需要网络/平台侧按同一时间窗提供：

### 8.1 Algorithm / protocol

Ring/Simple与Tree/Simple都在MCCL INFO中逐rank打印：

- `MCCL_ALGO set by environment to ...`
- `MCCL_PROTO set by environment to Simple`

并且benchmark有效完成，因此值没有被拒绝。证据层级是“变量被接受/读取”。
三种配置下日志都打印Trees和Channel拓扑，没有逐collective
`selected algorithm/protocol`字段；不能仅凭拓扑初始化日志断言实际AllReduce
一定执行Ring或Tree，也不能确认实际协议kernel。

### 8.2 Channel

固定4时：

- MIN/MAX变量均打印 `set by environment to 4`
- `Channel 00/04`
- `4 coll channels`

固定8时对应打印 `Channel 00/08` 和 `8 coll channels`。默认Ring/Simple为
32 coll channels。因此MIN/MAX_NCHANNELS的4与8在本环境中得到“实际生效”
级证据。

### 8.3 控制分类

- 生效：`MCCL_MIN_NCHANNELS=MCCL_MAX_NCHANNELS=4/8`
- 只被读取/接受：`MCCL_ALGO=Ring/Tree`、`MCCL_PROTO=Simple`
- 被拒绝：无
- 未知且未运行：`MCCL_IB_QPS_PER_CONNECTION`及其它channel/QP变量

QP没有合法值域、默认值和实际QP数量日志，因此按要求不运行。

### 8.4 解释边界

每配置只有1轮双节点能力门禁。带宽差异仅证明参数有行为影响，不能据此选择
生产修复或外推64节点；本轮没有改变TC/QoS，也没有扩大规模。
