## 4. Pilot 结果

指标语义：

- `avg_ms_global_max`：每轮取 64 rank 最慢完成时间，再对 20 轮求平均；
- `bus_bw_global_max`：按 AllReduce 的 NCCL-tests 同构折算因子，从上述平均时间计算；
- per-iter 分布：20 个逐轮 global-max 时间，不是各 rank local 时间混合分布。

| case | avg global-max (ms) | alg_bw (GB/s) | bus_bw (GB/s) | iter median (ms) | p95 (ms) | min–max (ms) | stdev (ms) |
|---|---:|---:|---:|---:|---:|---:|---:|
| 8n8r | 14.3832 | 18.6632 | 36.7431 | 8.2462 | 51.4280 | 7.3322–52.8926 | 13.8100 |
| 16n4r | 13.3120 | 20.1649 | 39.6996 | 12.9168 | 15.1527 | 12.7868–15.5122 | 0.8283 |

16n4r 相对 8n8r：

- global-max 平均时延：**降低 7.45%**
- bus_bw：**提高 8.05%**

但分布形状不同：

- 8n8r 的中位时延更低（8.25ms），同时存在约 51–53ms 尾部尖峰，导致均值显著升高；
- 16n4r 的中位时延更高（12.92ms），但 20 轮集中在约 12.79–15.51ms。

因此本轮不能简化为“节点更多一定更快”。更准确的 pilot 观察是：

> 8n8r 有更快的常态迭代，但伴随明显尾部尖峰；16n4r 常态迭代较慢却更稳定，最终 global-max 均值和 bus_bw 略优。

## 5. 计数器与结论边界

当前镜像没有本轮已验证、可读取的 RNIC / PFC / ECN 计数器入口，因此：

- 未采集计数器；
- 未伪造 queue / CNP / retransmit / PFC 数据；
- 无法把 8n8r 尾部尖峰直接归因到 fabric、RNIC、MCCL 或 GPU 调度。

这只是每 case **单轮 pilot**。不能作为最终统计结论，也不足以通过 G1 或排除背景噪声。

### 8.4 结论边界

该单轮形成一个强烈但待复现的信号：

> world 固定 64 时，节点数从 16 增至 32、每节点 rank 从 4 降至 2，没有继续改善；相反，32n2r 的常态 global-max 时延显著上移，带宽明显下降。

这更偏向“覆盖更多物理节点/路径或节点拓扑选择后出现新代价”，但仍不能区分：

- fabric 覆盖范围或故障域；
- MCCL 根据节点布局改变算法/channel/拓扑；
- 每节点仅使用 GPU 0、1 的 GPU↔NIC mapping；
- 背景噪声或单轮偶发状态。

当前无已验证计数器入口，因此未伪造 RNIC / PFC / ECN 证据。

本轮按约束停止：**不做 64×1，不重复 32×2**。下一步应先复核本单轮的 MCCL algorithm/channel/topology 日志策略，并由后续授权决定是重复 32×2 还是进入 64×1。

### 9.4 64n1r 的价值与风险（本轮未执行）

价值：

1. 补齐 world=64 matched-node 的极端点，判断 32→64 节点是否继续恶化；
2. 若 64n1r 继续下降，可加强“物理节点覆盖/路径或节点拓扑选择”候选；
3. 若 64n1r 回升或出现新跳变，可提示 GPU0 mapping、MCCL topology/algorithm/channel 或每节点进程布局不是单调关系。

风险与混淆：

1. 需要触及 64 个 pod，启动面和清理面扩大一倍；
2. 当前顺序发射 32 pod 已约 6 分钟，64 pod 预计约 12–15 分钟；启动偏斜和控制面偶发 EOF 风险更高；
3. 每节点只用 GPU0，会把“节点数”与固定 GPU0↔HCA 亲和绑定；
4. 仍无 RNIC/PFC/ECN 计数器，性能变化无法直接归因；
5. 单轮 64n1r 仍不能成为最终统计结论。

建议：**64n1r 的诊断价值已足够，但应作为单独授权的小步执行**。执行前应先把 64 pod 发射/清理、全局进展监控和 GPU0 mapping 快照写入 manifest；不应在本轮自动追加。

## 13. marker 状态机审计与修复（2026-07-18 21:06–21:12）

### 13.1 原协议问题

上一轮 start stage 同时承担了“提交 launcher”和“立即观察 torchrun”两种语义。64节点 rendezvous 约16秒后才进入 benchmark，但旧检查只等待约5秒，因此64节点全部误报 RC67，并触发 cleanup。

这不是通信 benchmark 的性能失败，而是 launcher 状态机错误。该轮：

- 64 rank JSONL 虽已写出；
- cleanup SIGTERM 与 teardown 重叠；
- marker 为 `40 done / 24 fail`；
- 全部结果保持 **INVALID**，禁止用于性能比较。

### 13.2 新 marker 协议

每个 `run_local`：

1. 启动后首先原子写 `.started`；
2. marker 包含 `RUN_ID / NODE_RANK / PID / timestamp`；
3. torchrun rc=0 原子写 `.done`，包含 `RC=0`；
4. 非零则原子写 `.fail`，包含实际 rc；
5. 每次运行前删除本 run 目录内同 rank 的旧 marker；
6. marker 读取必须精确匹配 `RUN_ID`，旧 run marker 不计数。

start exec 现在只表示 launcher 已提交：

- 原子写 pod `/tmp` pid state；
- pid state 包含 RUN_ID；
- retry 先检查同 RUN_ID pid，保证幂等；
- 不再因瞬时 pgrep 为空判失败。

cleanup 只匹配：

- 本 run 的 state RUN_ID；
- 本 run 的唯一端口和 AFS basename；
- 本 run launcher 内容。

### 13.3 统一等待状态机

新增 `scripts/cluster/launcher_state.sh`：

- 周期统计 started/done/fail/missing；
- launcher/torchrun live 只作为诊断，不以瞬时0直接失败；
- started或done增长刷新 `last_progress`；
- 明确fail立即失败；
- 只有 `now-last_progress >= timeout` 才超时；
- 64 done / 0 fail 才 COMPLETE。

本地测试覆盖：

- 延迟出现进程；
- 快速完成而 pgrep 已空；
- 部分 done 逐步增长；
- 明确 fail；
- 无进展超时；
- 旧 marker RUN_ID 隔离；
- 原子 marker；
- Bash 3.2。

结果：`test_launcher_state: OK`。
